# Linux Probing Notes

These notes capture what was verified directly against a connected Revelator IO 24 on Linux before Windows packet capture.

## Confirmed

- Device discovered by `lsusb` as `194f:0422`
- `PresonusDevice.find_devices()` returns the connected device
- Device can be opened from Python once udev permissions are fixed
- `get_device_info()` succeeds and returns:
  - vendor id `194f`
  - product id `0422`
  - product name `Revelator IO 24`
  - serial number present
- `get_descriptor(0x01)` returns an 18-byte device descriptor

## USB Layout Observed On Linux

- Configuration count: `1`
- Vendor-specific interface: `5`
- Vendor-specific alt setting with endpoints: `interface 5 alt 1`
- Vendor-specific endpoints:
  - `0x01 OUT`
  - `0x81 IN`

## Access Notes

- Initial blocker was `Access denied` until udev rules were installed
- After permissions were fixed, `set_configuration()` returned `Resource busy`
- Direct claim of interface `5` still worked
- Transport was updated to tolerate `Resource busy` on `set_configuration()` and continue
- Bulk reads from `0x81` timed out until interface `5` alt setting `1` was selected

## Bulk IN Behavior

After selecting `interface 5 alt 1`:

- Idle read from `0x81` returned 64 zero bytes
- The following provisional commands also returned 64 zero bytes after write:
  - mute on/off
  - solo on/off
  - phase on/off

Example transmitted packets during Linux probing:

- mute on: `02012501010001`
- mute off: `02012501010000`
- solo on: `02012601010001`
- solo off: `02012601010000`
- phase on: `02012701010001`
- phase off: `02012701010000`

## What This Does Not Prove

- It does not prove these command encodings are correct
- It does not prove the device interpreted the commands semantically
- It does not identify the true response format
- It does not reveal routing, DSP, preset, or state-query packet structure

## Conclusion

Linux probing confirms that:

- the device is reachable
- the vendor-specific interface can be claimed
- writes can be transmitted on the vendor-specific bulk endpoint

But authoritative protocol work still needs Windows packet capture so command and response structure can be verified against real control-app traffic.

## Controlled Linux usbmon Capture

A controlled Linux capture was taken while sending a known reversible command
sequence from `tools/send_known_sequence.py`.

Capture file:

- `captures/raw/revelator-sequence.pcapng`

Known transmitted sequence:

- mute on: `02012501010001`
- mute off: `02012501010000`
- solo on: `02012601010001`
- solo off: `02012601010000`
- phase on: `02012701010001`
- phase off: `02012701010000`

Observed in usbmon for device address `11`:

- frame `389/390`: host sets interface `5` alt setting `1`
- frame `391/392`: bulk OUT on `0x01` with `02012501010001`
- frame `395/396`: bulk OUT on `0x01` with `02012501010000`
- frame `397/398`: bulk OUT on `0x01` with `02012601010001`
- frame `399/400`: bulk OUT on `0x01` with `02012601010000`
- frame `401/402`: bulk OUT on `0x01` with `02012701010001`
- frame `403/404`: bulk OUT on `0x01` with `02012701010000`

What this proves:

- Linux usbmon can see our host-generated vendor traffic clearly
- The current transport path is using interface `5` alt setting `1`
- The active vendor bulk OUT endpoint is `0x01`

What this does not prove:

- that these command encodings are semantically correct
- that the device interprets them the same way Universal Control does
- that the all-zero or absent response behavior reflects the real control protocol

This capture is useful as a Linux-side baseline for later comparison against a
Windows capture generated by the official control software.
